# 色彩美学：Prism 的设计哲学

一种颜色，从来不是一个孤立的点。当你为产品选定一抹品牌色时，你其实已经在心里许下了一整套承诺：它的浅色背景该是什么模样，它在悬停时如何微微加深，它承托文字时是否还看得清，它在暗色界面里会不会显得浑浊而疲惫。Prism 的全部工作，就是把这一句许诺，兑现成一套完整、和谐、对比度可靠的设计令牌系统。

我们相信：**给定一种颜色，就已经隐含了一整个系统。** 设计师真正需要的不是一支取色笔，而是一位懂得色彩科学、并愿意为每一步负责的同伴。

## 感知均匀，而非数学均匀

大多数调色板生成器在数值上很"整齐"——明度等差、色相等分、饱和度线性递减。但人眼并不是按数学刻度感知颜色的。在 HSL 里把明度从 50% 降到 40%，与从 90% 降到 80%，数字上等距，视觉上却天差地别;中间调的一点点变化，远比两端醒目。

Prism 拒绝这种"看上去公平"的均匀。我们的明度阶是**手工调校、按感知间隔排布**的——它不是一条直线,而是一条在中间调放大跳变、在两端收敛的曲线。我们在 OKLCH 这个感知均匀的色彩空间里工作(详见 [为什么是 OKLCH](./oklch.md)),让"等距"真正等于"看起来等距"。

## 品牌色是神圣的

许多系统会先把你的品牌色"洗"进它们的算法里,吐出来的色板中,你最初选定的那一抹颜色已经面目全非。Prism 不这么做。

你输入的精确颜色会被**逐字保留**,钉在它按明度最接近的那一级台阶上(借鉴 Radix 的 remainInput 思想)。因为整条明度阶是单调的,种子色永远落在它两个邻居之间,既不会破坏单调性,也不会被算法篡改。你交给我们的颜色,会原封不动地出现在成品里。

```ts
import { generateTheme, toHex } from "@prism/core";

const theme = generateTheme({ primary: "#3b5bdb" });

// 种子色被钉在最接近的台阶上,逐字保留
console.log(theme.seeds.primary); // 你输入的精确颜色
console.log(toHex(theme.scales.primary["500"])); // solid 锚点
```

## 调和是算出来的,不是猜出来的

配色不该是一场赌博。次要色、第三色、中性色、语义色——它们与品牌色之间的关系,是有结构、可计算的。Prism 用真正的色彩调和规则推导它们:次要色取分裂互补(色相 +150°),第三色按 Material 规则取 +60°,而后对每一次色相旋转都执行同一条铁律——**重新对齐明度,重新把彩度收敛到共享目标**。于是新色与品牌色像是同一家族的兄弟姐妹,而不是硬生生贴上去的补丁(详见 [调和](./harmony.md))。

## 对比度是求解出来的,不是祈祷来的

"这个文字颜色在这个背景上应该够清楚吧?"——这是一句祈祷,不是一项工程。

Prism 把可读性当作约束来求解。我们以 APCA(SA98G)为生成目标,用 `solveForLc` 在明度轴上二分搜索,直到对比度精确命中目标值,再以 WCAG 2.x 作合规性报告。无论你的品牌色是什么色相,正文文字、强对比文字、以及叠在 solid 色块上的前景色,都被解到位——这正是 HSV 染色系统(如 AntD/Arco)留下的缺口(详见 [对比度](./contrast.md))。

## 一种世界观

把这些信念合起来,就是 Prism 的世界观:

- 颜色是系统性的,不是点状的;
- 均匀要服从感知,不是服从算术;
- 品牌色不可侵犯;
- 和谐可以计算;
- 可读性必须求解。

::: tip 一句话
你只需诚实地选好一种颜色。剩下的色彩科学,交给 Prism。
:::
